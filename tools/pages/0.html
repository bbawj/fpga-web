# Hosting a micro blog on an FPGA I wanted to learn digital logic design. Here are my criteria: - Comical and somewhat impractical. Makes it fun, low stakes, and novel. - Make use of and generate tangible IO. Networking serves this need the best, with standardized protocols and tools whose effects can be readily observed. - Cheap and or basically free development process. open source tools have really came through to enable this project to even be possible. After some deliberation it hit me. You are reading this right now on my blog. Why not serve you the content directly from an FPGA. There are many reasons why people dont usually do this, which conveniently satisfies criteria 1. To not die by scope creep, I landed on bare HTTP. This way, we can ignore TLS and encryption. This also means that we would have to make use of a reverse proxy to perform this task. This is quite a common network architecture, and is also the one used in my home network. This also allows us to make simplifying assumptions on the way to the final goal. More on this later. # Hardware choice the color light was an easy and natural choice. there aren't many FPGA boards that are cheap and also come with on board Ethernet PHY. On top of that, it even has 2 PHYs, 2MB of SDRAM and flash storage. Others have provided a more concrete run-down as well as reverse engineering of the pins. Link to where you can buy one ![here](). The setup used for development is a simple point to point Ethernet connection with my laptop. We can even repurpose one of the IO pins as a UART TX line. This would serve as our mode of printf debugging for the entire project. ![the hardware setup](setup.png) # ARP: The first stepping stone The first key milestone is responding to Address Resolution Protocol (ARP) packets. These are the baseline, and requires the device have an IP and MAC address to respond with. Easily enough, we can hardcode this. ``` localparam [47:0] LOC_MAC_ADDR = 48'hDEADBEEFCAFE; localparam [31:0] LOC_IP_ADDR = 32'h69696969; // 105.105.105.105 ``` One available tool in Linux is arping which let's you specify the target MAC address and craft a ARP request rather than IPV4 ICMP packet. Getting this to work would mean successfully driving the RGMII interface of the PHY. The design I decided to go with is to process the frame data in-place rather than saving the entire frame in a buffer and processing it separately. This means the decoding logic runs off the PHY RXC clock. There are a few advantages of this approach. The first is slightly reduced latency as the decoding is done during the receive itself. At 1 gigabit, we would save at most max frame size / 1 gigabit seconds. Another one is that a separate buffer for the frame is not necessary. However, this means the decoding logic must also keep up with RXC. This is fine as we are mostly selecting parts of the packet based on a counter value or state machine. The checksum calculation can also be done per-cycle every 2 bytes. With that, the next thing to do is generate ARP replies. From the receive path, info such as the source hardware address and IP has to be extracted. The main challenge here is to do a Clock Domain Crossing from the PHY RXC, into the system clock. This is a solved problem using a gray-coded counter FIFO, and one of the main references for doing this is Clifford Cumming's paper. Rather than crossing all the decoded values, only the done and error signals are crossed and then used to trigger the transmit logic. To keep the transmit path simple from the start, I chose to stream the data from the encoder directly to the RGMII lanes, rather than to first build a packet in a separate buffer. The main issue with this approach is for larger packets. Since each transmission has to obey the Ethernet maximum segment size of 1500 bytes, larger payloads will eventually need to be split into multiple packets. This would come back to bite me later, but works OK for ARP. The main challenge here is timing. Driving a 1gigabit RGMII lane first requires a 125 mhz clock. the color light offers a 25mhz clock generated externally by the PHY. We can quite easily create the frequency we need using the EHXPLL primitive. Thankfully, a good reference can be found in projecttrellis. They have even documentated examples of which parameters to change for generating more complex clock signals. Specifically, the RGMII spec requires a minimum setup and hold time of 1.2ns of on the data and control signals. The solution I came up with, which is also what is described in this excellent post by ![etherealwake](https://etherealwake.com/2025/03/ethernet-rgmii/), is to generate the data on one clock, while the TXC is offset by 90 degrees with respect to that clock. I found this to work well in practice. One important note is to ensure that the method of doing so creates synchronized clocks. This means using 1 PLL structure and ensuring no combinatorial delays between them. And voila WIth ARP support, any device can now discover the IP address of our FPGA. In fact, if we probe the interface in wireshark we can see that this is done by the host before the TCP handshake is initiated. In the next section, I describe how to achieve the "hello, world" of networking tutorials, a TCP echo server on the FPGA. # Getting to a TCP Echo Server For a bare minimum TCP echo server, we can strip down most of the components that make TCP challenging. The big win in terms of complexity is to ignore out of order packets, TCP's main objective to provide reliable delivery of packets. We forgo this property by assuming the network is controlled with no chance of re-ordering and leave this for future me. For the final set-up, since the request is mediated by our reverse proxy anyway, the re-ordering is limited to the network path from the reverse proxy to the FPGA. Hence, for our use case, packets received out of order (OOP) are silently dropped. Basically, if the sequence number received is not expected, the TCP engine will reject the payload and no ACK is sent. This would rely on the sender to detect this packet as lost and re-transmit the entire payload once more, hopefully allowing the packet to be received in-order at the FPGA. However, this is definitely not the right solution, and only solves issues where the OOP is a result of some intermittent issue on the network path like a spike in traffic. It could be that OOP are perpetually received, because out of order is not a pathological thing (i.e. network placement, routing placement). Since we are self-hosting this, and testing is done peer-to-peer, we can safely ignore this for now. ## Core 2 main data structures are used Packet: Contains the Information decoded or required for encoding onto the RGMII. This is metadata, such as the IP addresses, port, payload size, address to the payload in memory. Transmission Control Block (TCB): represents one tcp connection and holds all the state for it. The basic metadata is stored in on-chip registers and is used often by the engine to check for valid packets and create new ones. ``` typedef struct packed { // to identify this TCB uniquely logic [31:0] peer_addr; logic [15:0] peer_port; // Sequence number for transmitting our own data. It is the sequence_num // for the next transmit packet logic [31:0] sequence_num; // Ack number used for transmitting. i.e. the number expected by the // receiver (received sequence_num + 1) logic [31:0] ack_num; logic [15:0] window; CONN_STATE state; } tcb_t; // in registers tcp::tcb_t tcb_mem; ``` In addition, each TCB module holds 2 other FIFOs. - to_send is a FIFO holding pending transmit packets. These are written by the arbiter in the next section when upper layer logic wants to make a transmission. - to_ack is an internal fifo holding packets that have been submitted to the transmission engine but waiting for an ack response. An ACK received will clear this FIFO. An ACK received will clear this FIFO. Cumulative ACKs (ack number is larger than the smallest ACK number we have in the FIFO) are easily supported; by popping off from the FIFO until it is either empty, or we reach the packet the ACK is meant for). Although I previously mentioned how OOP are ignored, this is only safe on the RX path. On the TX side, it is definitely likely the packet gets lost in the network towards the client. To support this, a retransmit timer would sit here and copy the packet metadata back into the to_send FIFO. ## Arbiter The number of simulatenous active connections is limited by the number of TCBs. To handle multiple TCBs, an arbiter is used. It activates the correct TCB based on the IP.address and port within the packet data. It also checks which TCBs have a pending packet to send and is in charge of muxing these requests with the transmission module. It also arbitrates access to the TCP state machine handler. There are 3 kinds of requests that the arbiter needs to handle. 1. Echo request. This is our MVP. It bypasses 2. State machine requests. State Machine The original state machine is rather large. For an echo server, we need only implement the responder sections. We can also remove close-wait by sending fin and ack together. The entire thing is implemented as a module separate from the arbiter. Its inputs are the packet data such as sequence number etc. whether the packet data is intended for the RX or TX path, the current TCB data such as the current sequence number and uses 1 clock cycle to determine the validity of the packet data. It's outputs determine how the TCB should update its current state e.g. increment sequence number. Before going further, here is a block diagram for all of what's discussed: ![block diagram](https://github.com/bbawj/fpga-web/blob/main/docs/FPGA-web-archi.svg) # A completely static HTTP In the process of achieving the TCP echo server, we have almost all the required components for serving a webpage; instead of echoing the payload, we just need to parse for a semi-valid HTTP request and respond accordingly. Crucially for this application, we do not need to care about independently triggering any payloads from the FPGA as we only respond to what was sent to us. This means that there are no race conditions to deal with regarding state updates due to parallel RX and TX paths. They are all serialized as long our processing can complete within the 96bit times of inter-packet gap enforced by the Ethernet standard and we buffer enough of the necessary state. The HTTP decoder is simple enough, read the first line for the method and target, ignore all headers. We just need a way to translate the endpoint to the data for the page. Minimally, this means the starting address of the content in memory as well as the size. The 4MB of non-volatile flash memory will hold all the static pages, written once during programming. This way, I could also add additional pages in the future by writing to flash. However, reading from flash during runtime would also be prohibitively slow, which means a SDRAM controller is in-order. There is actually slightly more to keep things completely static. For this first approach that builds the packet in-situ during the transmission, not only must the content be static, but the payload checksum needs to be precomputed. Moreover, the entire HTTP header is statically stored together with the content. You can start to see how this breaks down if we have to support packets larger than max segment size (although technically if the reverse proxy is configured for jumbo segments, this would eliminate the problem). Putting everything together, the memory flow for the system would be: 1. Navigate to page URL endpoint, for example "/1" 2. Use endpoint as a key to lookup the actual page content starting address in SDRAM 3. Use endpoint as key to lookup the size of the page content in block RAM 4. Use endpoint as key to lookup the precomputed checksums for the content. Due to the static nature of each page, this makes sure the TCP header checksum can be computed during transmission in-situ. 5. Transfer page content from SDRAM into block RAM and transmit ## Dealing with memories The SDRAM controller is a dual port interface to a single port memory, and provides grant signals to indicate that the request has been latched into the state machine. The addressing width is according to what the hardware expects, 11 bit row and 8 bit column. module sdram_ctrl #( parameter int FREQ = 125_000_000 ) ( input clk, input rst, input wr_req, // 18:8 is the row address, 7:0 is column address input [18:0] wr_ad, input [31:0] wr_data, output reg wr_granted, input rd_req, input [18:0] rd_ad, output reg rd_valid, output reg [31:0] rd_data, output reg rd_granted, // Boot complete signal. 1: boot done output reg boot_done, output reg [1:0] sdram_ba, output reg sdram_we_n, output reg sdram_cas_n, output reg sdram_ras_n, output wire sdram_clk, inout [31:0] sdram_dq, output reg [10:0] sdram_addr ); One other detail is the tristate nature of sdram_dq. Luckily yosys seems to support this alright. I added additional constraints to force the place and route tool to conform: IOBUF PORT "sdram_dq[0]" IO_TYPE=LVCMOS33; IOBUF PORT "sdram_dq[1]" IO_TYPE=LVCMOS33; // ... I designed an initialization module that would pull all the pages into SDRAM once reset is released and the SDRAM initialization is complete. (Since SDRAM is only 2MB, and I only use 1 bank for now, the total memory limitation must be under 512KB). Page-content cache I add-on 18kb block ram as a cache for page content which would help avoid some trips to main memory. One more added constraint to make things easier for me is that each page must fit inside the cache and hence the max size per page is 18kb. All in all, this basic setup should support at least 28 18kb pages which is plenty for a static site. To manage the page-content cache, a basic slab allocator is placed in front. A slab allocator provides sequential allocation access to the memory range, and deallocation means deallocating the entire chunk. It takes the request size, adds that to the current used memory counter, and checks if it fits. If it fits, the address is returned. Otherwise, it asserts an error signal and the requester must use the rst signal to put the counter back to 0. With the self-imposed memory limitations, there can be at least 28 pages if each is 18kb, 512 pages if each is 1kb and so on. Cache structure 18kb cache: size of 1 block ram cell, 512 slots, addressable with 10 bits. 32 bit blocks: min tcp payload pad to 4 bytes and interestingly, also the min size of a valid HTTP page (ab) So, using the physical memory (SDRAM) address, the first 10 bits lookup the offset in cache. They also lookup in parallel a separate RAM block of tags for comparison with the remaining 9 upper bits of the address. Valid bits are also stored for each slot for invalidating the entry if the slab allocator is reset. One challenge when designing all this is meeting timing @ 125 mhz. The clk-to-q delay for reads is already around 5.8ns. Hence, any logic using these read values should be registered. Unfortunately, my first attempt did not synthesize to a registered output. ``` always @(posedge clk) begin data <= mem[addr]; match <= data == tag; // seemed to fold into mem[addr] == tag end ``` Using (*keep*) or (*dont_touch*) attributes on the registers also did not seem to work. Adding a pipelined register manually also did not seem to break up the path. What worked in the end is to manually instantiate the DP16KD/PDPW16KD primitives, with the OUTREG REGMODE parameter. Luckily, this parameter is supported by the downstream tools, updating clk-to-q to 0.5ns. There were also other frustrations. OCER, are the clock enables for those output registers and should be tied accordingly for OUTREG to work. I also wanted to support asymmetric read and write widths such as for streaming quickly at 32 bits/cycle but then reading narrower at 8bits/cycle for transmitting. The primitives also support this natively, but care is needed to make sure the output and input port connections are correct. It took a while, but making use of the native Lattice IPExpress tool to generate the port connections allowed me to figure out what went wrong. An example below: ``` PDPW16KD #( .REGMODE(REGMODE), .DATA_WIDTH_R(EBR_WIDTH_B), .DATA_WIDTH_W(EBR_WIDTH_A) ) ebr ( .DI0 (dia[0]), .DI1 (dia[1]), .DI2 (dia[2]), .DI3 (dia[3]), .DI4 (dia[4]), .DI5 (dia[5]), .DI6 (dia[6]), .DI7 (dia[7]), .DI8 ('0), // this killed me .DI9 (dia[8]), //... .DI17 ('0), // why didnt i check the tool earlier .DI18 (dia[16]), //... .DI26 ('0), //... .DI35 ('0), ``` ## Edge cases With that, opening the IP and port on a browser is able to display the endpoint properly. However, as it currently is, refreshing the page will cause the page to load for a long time. This is because in HTTP 1.1, browsers make use of a persistent keep-alive TCP connection to avoid needing to restart the 3-way handshake, and do not automatically terminate after each request (by sending FIN). Instead, a new TCP connection is created and often, parallel connections are started to avoid head of line blocking for multiple resources. For e.g. even though my site does not specify any favicon, a get request for it is made from Chrome browsers. In HTTP 2, browsers will make use of 1 TCP connection, and multiplex streams on the HTTP layer with a stream ID. However, since we are on very limited hardware, we would prefer if every request is terminated swiftly, so that someone else might reach the FPGA. Introducing HTTP 1.0! Responding with that in the header would cause the browser to send a FIN immediately once it receives a response. Even if it does not, I designed each TCB to transition to the TCP FIN-WAIT state once it has transmitted the payload, also tagging each response with the FIN flag to aggressively close any connections. Hence, the main way that clients can hog the TCB is if no request is made, or a request that is not HTTP is made. An idle timer is introduced to transition the connection to FIN in these cases. This is all very bad for performance reasons, but is a trade off for working with multiple clients with only a few TCBs. # This project has no need to exist All of this has come together so that typing in http://105.105.105.105:8080/0 into a browser would serve the following page: This is no big feat.
